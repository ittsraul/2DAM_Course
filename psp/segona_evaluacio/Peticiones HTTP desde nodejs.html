1 Introducción. Comunicación entre aplicaciones.

Las Apis RESTFul (Representational State Transfer o Transferencia de Estado Representacional) son una arquitectura de software orientada a recursos (ROA) para implementar web services. Se trata de una arquitectura moderna y cada vez más popular, que hace uso del protocolo HTTP para transferencia de datos, y generalmente el formato JSON para su representación. Así pues, con REST podremos crear aplicaciones y servicios entre varios dispositivos, siempre y cuando estas entiendan el protocolo HTTP, siendo Internet el medio natural a través del cual ofrecer estos.

Cómo hemos dicho, REST presenta una arquitectura orientada a recursos, donde cada recurso del servidor dispone de una ruta (dirección URL) diferente, conocida también como endpoint. Estos recursos se procesan con los métodos de la interfaz uniforme GET, POST, PUT y DELETE (HTTP tiene más métodos, pero no pertenecen a esta interfaz).

El API que vamos a utilizar para los ejemplos se encuentra documentada en el siguiente repositorio de Github. Para acceder a los recursos, podemos probar con el navegador web las direcciones que en ella se especifican, y que se encuentran alojadas al servicio Railway App.

Por ejemplo, puedes probar desde tu navegador la siguiente dirección URL:

  • https://node-comarques-rest-server-production.up.railway.app/api/provincies

Cuando hablamos de web services debemos mencionar el protocolo SOAP. Dicho protocolo, fue diseñado antes que REST, y definía la forma en que diferentes plataformas y lenguajes podían intercambiar datos de un modo sencillo, haciendo uso de XML para el intercambio de información.

Más adelante y en otros módulos profundizaremos en este tipo de servicios.

2 Peticiones HTTP

Para poder realizar peticiones web, ya sea a servicios REST o cualquier otro recurso que se encuentre en la web existen diferentes librerías que podemos usar en nuestras aplicaciones javascript.

Cuando trabajamos desde Javascript para el navegador, éste incorpora el objeto XMLHTTPRequest, que supone la base sobre la que se construyen aplicaciones AJAX (Asynchronous Javacript XML). Por su parte, nodejs incorpora de manera estándard los módulos http y https, que nos permiten tanto crear servidores HTTP/HTTPS como realizar peticiones de este tipo.

El esquema básico de una petición de tipo GET para obtener un recurso web sería el siguiente:

const https = require('https');

https.get(URL, (res) => {
    // Variable donde guardar los datos recibidos
    let data = ''; 

    // Evento que se dispara cada vez que se reciben 
    // fragmentos de datos.
    res.on('data', (chunk) => {
        data += chunk;
    });

    // Evento que se dispara cuando se ha recibido
    // toda la respuesta.
    res.on('end', () => {
        console.log(JSON.parse(data).explanation);
    });

// Evento que se dispara en caso de error.
}).on("error", (err) => {
  console.log("Error: " + err.message);
});

La mayoría de funcionalidades del módulo HTTP y HTTPs son de bastante bajo nivel. En este caso, recibimos la respuesta a nuestas peticiones en fragmentos de datos que debemos analizar y reconstruir. Estos fragmentos son bastante sencillos de reconstruir cuando se trata de respuestas en formato JSON, pero no deja de suponer un paso adicional.

Analicemos por partes qué hemos hecho:

  • En primer lugar, importamos el objeto https del módulo con el mismo nombre:
const https = require('https');
  • Con este objeto, invocamos a su método get con el siguiente código:
https.get(URL, (res) => {
    // Cuerpo del callback    

    }).on("error", (err) => {
        console.log("Error: " + err.message);
    });

Como vemos, al método GET le proporcionamos la URL o recurso al que deseamos acceder, así como una función de callback, que recibe el parámetro res con la respuesta. Si se produce alguna excepción en esta llamada, se ejecuta el segundo fragmento de código, mostrando el error por pantalla.

  • Dentro del cuerpo del callback que procesa las respuestas, lo que hacemos es preparar una estructura para almacenar datos cuando son recibidos:
let data = ''; 

res.on('data', (chunk) => {
    data += chunk;
});
  • Y cuano terminamos de leer datos, lo que hacemos es ya procesar la respuesta completa, en este caso, mostrarla por pantalla:
res.on('end', () => {
    console.log(JSON.parse(data).explanation);
 });

Veamos cómo aplicaríamos este esquema para obtener los datos sobre provincias:

import { get } from 'https';

get('https://node-comarques-rest-server-production.up.railway.app/api/comarques', res => {
    let data = '';

    // Obtenemos el código de estado de la respuesta (200=Ok)
    console.log('Status Code:', res.statusCode);
    // Y la fecha de ésta
    const headerDate = res.headers && res.headers.date ? res.headers.date : 'no response date';
    console.log('Date in Response header:', headerDate);

    res.on('data', chunk => { data += chunk; });

    res.on('end', () => { console.log(JSON.parse(data)); }
    );
}).on('error', err => {
    console.log('Error: ', err.message);
});

2.1 Concatenando peticiones

Ahora bien.. si ahora quisiéramos para cada provincia obtener el nombre de las diferentes comarcas, qué haríamos?

const https = require('https');

https.get('https://node-comarques-rest-server-production.up.railway.app/api/comarques', res => {
    let data = '';
    res.on('data', chunk => {
        data += chunk;
    });
    res.on('end', () => {

        let provincies = JSON.parse(data);
        
        for (let provincia of provincies) {
            console.log('https://node-comarques-rest-server-production.up.railway.app/api/comarques/' + provincia.provincia);
            https.get('https://node-comarques-rest-server-production.up.railway.app/api/comarques/' + provincia.provincia, res => {
                let data = '';
                res.on('data', chunk => { data += chunk; });

                res.on('end', () => {
                    let comarques = JSON.parse(data);

                    console.log(" \nComarques de " + provincia.provincia + "\n");
                    for (comarca of comarques) {
                        console.log(comarca);
                    }
                });
            }).on('error', err => {
                console.log('Error: ', err.message);
            });

        }
    });
}).on('error', err => {
    console.log('Error: ', err.message);
});

Como vemos, cuando se finaliza la recepión de las provincias, debemos recorrer éstas, y realizar de nuevo peticiones, que se procesan del mismo modo, añadiendo un nuevo nivel de anidamiento de callbacks.

Disponéis de más información sobre transacciones HTTP en los siguientes enlaces:

  • https://nodejs.org/api/http.html

  • https://nodejs.org/en/docs/guides/anatomy-of-an-http-transaction/

3 Peticiones web con fetch

El API fetch se introdujo en la versión 6 de EcmaScript, y permite realizar peticiones asíncronas mediante Promesas, que también se introdujeron en esta versión. Mediante promesas vamos a poder trabajar con peticiones asíncronas de una manera más cómoda que con los callbacks.

Esta funcionalidad se implementó de forma experimental en el core de nodejs en la versión 17.5. Hasta ese momento, se requería de librerías como Axios o node-fetch.

La forma de realizar una petición con fetch es relativemente sencilla, y consiste en invocar a fetch proporcionándole como argumento la URL del recurso que deseamos obtener. Por ejemplo:

const res = fetch('https://node-comarques-rest-server-production.up.railway.app/api/comarques');

El API fetch devuelve un objeto de tipo Promise con sus dos métodos then() y catch(). Estos métodos reciben sendas funciones de callback que serán invocadas bien cuando se obtenga respuesta o bien cuando se produzca un error. Hay que tener en cuenta que los erros que harán que se ejecute el catch son aquellos relacionados con errores de red o de otro tipo, pero no los errores HTTP que devuelven un código de error (tipo 404, etc). Estos errores se tendrán que tratar dentro del then().

res.then(function(result) {
    // Procesamiento de la respuesta
    })
 .catch(function(err) {
    // Tratamiento de errores de red
 });

O usando funciones flecha:

res.then((result) => {
    // Procesamiento de la respuesta
    })
 .catch((err) => {
    // Tratamiento de errores de red
 });

También hay que tener en cuenta que lo más habitual será encontrarnos con dos promesas encadenadas para acceder al cuerpo del mensaje que nos devuelve el servidor. Veámoslo con un ejemplo. Para el caso de obtener las provincias, si mostramos el resultado por pantalla:

const res = fetch('https://node-comarques-rest-server-production.up.railway.app/api/comarques/provincies');

res.then((result) => {
    console.log(result);
});

Nos imprimirá el JSON con todo el objeto Response devuelto por el servidor:

Este objeto de tipo Response que se nos devuelve, no dispone del contenido del cuerpo (body) de forma directa, sino que proporciona diferentes métodos para acceder al contenido, como son .json() para obtener el contenido en formato JSON o bien .text(), para obtenerlo en formato texto. Estos método .json() y .text(), nos devuelven otra promesa, que se resolverá cuando se haya obtenido el contenido en uno u otro formato. Esta promesa será tratada con otro then encadenado, por ejemplo, para mostrar el resultado:

 const res = fetch('https://node-comarques-rest-server-production.up.railway.app/api/comarques/provincies');

res.then((result) => {
    // Obtenemos el contenido en JSON
    return (result.json());
}).then((result) => {
    // Mostramos el JSON por pantalla
    console.log(result);
});

4 async/await

En el apartado anterior ya introdujimos la mejora sintáctica async/await de EcmaScript 2017. Esta misma sintaxis podemos utilizarla para procesar las promesas que recibimos cuando usamos la API Fetch.

El ejemplo de obtención de las provincias sería:

async function obtrenirProvincies() {
    let resposta = await fetch('https://node-comarques-rest-server-production.up.railway.app/api/comarques/provincies');
    let respostaJSon = await resposta.json();
    console.log(respostaJSon);
}

obtrenirProvincies();
